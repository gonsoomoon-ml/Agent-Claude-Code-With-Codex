# ④ Web UI — v1.1c 체험 진행 UX (A+B) 설계 스펙 (2026-06-28, LANE B)

> 증분: v1.1a 체험하기(LIVE)의 UX 보강. 문제: 체험하기 클릭 후 페이지가 정지처럼 보임 — 비동기 생성(몇 분)에 피드백 0.
> 사용자 결정: **A(즉시 피드백) + B(실시간 진행)** 둘 다.

## 0. 결정
- **A** — 버튼 loading state · 작은 `<p>` → 눈에 띄는 **상태 카드** · 스피너 + 경과 타이머 · 명확한 기대치/안내.
- **B** — runtime mode=trial 이 **단계별 status 를 briefing-trials 에 기록** → `GET /trial/status?email=` → 프론트 **폴링**으로 카드가 "생성 중 → ✅ 발송됨"까지 따라감.
- 구독하기도 같은 패턴(loading + 상태 카드; putProfile 동기라 폴링 불요).

## 1. 상태 lifecycle (briefing-trials, PK=email)
`verification_pending`(새 이메일, POST 시) | `sending`→`generating`(검증됨/검증완료, 생성 시작) → 단말: `sent`(+`published` N) | `fallback`(published==0) | `expired`(검증 타임아웃) | `failed`(에러).
- 기존 `status` 필드 재사용(+ 신규 `published`(N) · `updated_at`). `status` 는 **DDB 예약어** → ExpressionAttributeNames `#s` 우회.

## 2. 컴포넌트

### 2a. runtime — `_trial.run_trial` 에 `status_fn` 주입 (DI)
- `run_trial(..., status_fn, ...)`: poll 실패→`status_fn("expired")` · 검증 후→`status_fn("generating")` · published>0→deliver→`status_fn("sent", published)` · ==0→fallback→`status_fn("fallback")`.
- entrypoint(`agentcore_runtime.py` mode=trial bg): `status_fn` = briefing-trials `update_item`(Key=email, SET `#s`=status [+published]) 클로저. except→`status_fn("failed")`. dry_run→status_fn=no-op.
- 검증대기 단계: 새 이메일은 POST 가 이미 `verification_pending` 세팅 → 검증되면 `generating` 으로 전이.

### 2b. api — `GET /trial/status?email=` (app.py, public)
- `validate` email 형식 → `TrialStore.get_status(email)` → `{status, published?}`. 행 없으면 `{status:"unknown"}`.
- `TrialStore.get_status(email)`(webapi/trial.py): briefing-trials get_item → status·published.
- ⚠️ 이메일-키 조회 = 임의 이메일의 *상태*(내용 아님) 노출 = 경미한 leak — 데모 수용, defer(trial_id 토큰화).

### 2c. web — 체험하기 진행 UI (Form.tsx + api.ts)
- `api.ts`: `getTrialStatus(email) -> {status, published?}`.
- 체험하기 onClick: 버튼 disable+"보내는 중…" → postTrial → 202 면 **상태 카드** + **3초 폴링** 시작.
- **상태 카드**(눈에 띄는 styled): 스피너 + **경과 타이머**(0:48) + status별 카피:
  - `verification_pending` → "📧 확인 메일의 링크를 클릭하세요 (받은편지함/정크)."
  - `generating` → "⏳ 브리핑 생성 중… {타이머} (보통 1~5분). 창은 닫으셔도 됩니다."
  - `sent` → "✅ 발송 완료! {published}건 — 받은편지함을 확인하세요."
  - `fallback` → "오늘은 새 소식이 부족해 안내 메일을 보냈어요."
  - `expired` → "확인 시간이 지났어요 — 다시 시도해주세요."
  - `failed`/error → "문제가 생겼어요 — 다시 시도해주세요."
- 폴링: 3초 간격, **단말 상태(sent/fallback/expired/failed)에서 정지** + 전체 타임아웃 ~10분(이후 "메일을 확인해보세요"로 정지).
- 구독하기: 동일 버튼 loading state + 상태 카드(putProfile 동기 응답 delivery=active/verification_pending → 카드 표시; 폴링 없음).
- 상태 카드 컴포넌트 분리(`StatusCard`) — 체험/구독 공용.

## 3. 테스트
- **runtime**(pytest): `run_trial` 가 status_fn 을 올바른 순서로 호출(generating→sent/fallback, timeout→expired). fake status_fn 기록 검증.
- **api**(pytest): `GET /trial/status?email=` (행 있음→status, 없음→unknown) · `TrialStore.get_status`. fake table.
- **web**(vitest): `getTrialStatus` payload · 폴링이 단말 상태에서 멈춤 · 상태별 카피 매핑 · 버튼 loading state.
- **e2e(moongons)**: 체험 → 카드가 generating→sent 로 갱신(캐시히트면 빠르게 sent).

## 4. 배포 순서
1. runtime(_trial status_fn + entrypoint writer) → **재배포**.
2. api(GET /trial/status + TrialStore.get_status) → deploy_api 재배포.
3. web(StatusCard + 폴링 + 버튼 loading) → deploy_web 재배포.
4. e2e(moongons).
