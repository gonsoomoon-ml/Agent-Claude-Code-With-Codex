> **상태: DRAFT (미구현·미배포).** 하드 선행조건(certifier 산술 정규화)이 충족되기 전에는 머지 금지 — 아래 §5.
> 근거 = 35 에이전트 리서치 워크플로우 + 프로덕션 카드 223건(13일) 감사 + 독립 재현 측정.

## 1. 문제 — 길이가 아니라 lead bias

발단은 "요약 길이가 기사마다 들쭉날쭉하다"는 관찰이었으나, 데이터는 **다른 병**을 가리켰다.

| 측정(원문 1500자+ 카드 101장, 독립 재현) | 값 |
|---|---|
| 요약이 원문에서 도달한 최심 위치 | median **0.614** |
| 같은 카드·같은 호출의 claims 가 도달한 최심 위치 | median **0.905** |
| 요약이 claims 보다 앞쪽에 머문 카드 | **77%** (78/101) |
| 요약이 원문 중간(0.5)조차 못 넘은 카드 | **39%** (39/101) |
| corr(요약 길이, 도달 최심) | **+0.47** |

**해석:** author 는 기사를 다 읽었고 다 이해했다(claims 가 기사 끝까지 정확히 도달). **summary 필드에서만 그 이해를 버린다 — 지시하지 않았기 때문이다.** 능력 실패가 아니라 **사양(specification) 실패**다.

부수 결함:
- **논조 역전** — 헤지 마커를 가진 claims 59장 중 **32장(54%)**의 요약에 유보 표현이 전무. 비판 기사가 홍보 요약이 되어 '✓ 검증' 배지를 달고 나간다.
- **층 책임 역전** — **44%(46/104)** 카드에서 why(해석층)가 summary(사실층)에 없는 hard fact 를 나른다.

## 2. 왜 프롬프트가 유일한 방어선인가 (아키텍처 근거)

SummEval 은 요약 품질을 **Consistency**(쓴 것이 원문에 함의되는가)와 **Relevance**(중요한 것을 골랐는가)로 분리한다. 우리 구조에서:

- certifier 는 `Envelope` 4필드(`source_excerpt`·`claim_text`·`claim_type`·`schema`)만 본다 → **Consistency 만 검사 가능**. "빠진 게 있나"는 원리적으로 볼 수 없다(최소 컨텍스트 = decorrelation 의 대가이지 결함이 아니다).
- **summary 는 certifier 에 아예 가지 않는다** — 검증되는 것은 claims 뿐(verdict 2,438건 중 summary 겨냥 0건).

→ **대표성은 author 생성 계약이 유일한 방어선이다.** 따라서 "게이트에 대표성 검사를 추가하자"는 후속 제안은 **기각** — 누락 판정에는 원문 전체 + author 의 선별 맥락이 필요한데 certifier 에 그것을 주면 decorrelation 이 조용히 깨진다.

## 3. 기각된 처방 (재제안 금지)

| 처방 | 기각 사유 |
|---|---|
| 문장 수 밴드('2~5문장') | **no-op** — 이미 91%(95/104) 준수, 구속 8장. 밴드 안 96장의 길이가 95~705자(7.4배)로 산포 100% 생존 |
| 자 수 목표('250자 내외') | 문헌 exact match 7.0%(문자)·0.8%(토큰). Anthropic structured outputs 는 문자열 길이 제약 미지원 + 우리는 `claude -p` CLI 라 애초 불가 |
| 총 자수 cap(400·500자) | **대리지표 함정** — cap 이 자를 20장이 커버리지 최상위군(최심 0.689 vs 나머지 0.449). corr(자수,사실개수)=+0.68 → 긴 카드는 커버해서 긴 것 |
| 원문 비례 / 압축률 계약 | 인간 편집자 스케일링 지수 k=0.147(CNN/DM)·0.072(XSum) = 사실상 고정 길이. **우리는 이미 k≈0.337 로 인간보다 2배 반응 중**이고, 모델은 원문 크기가 아니라 내용량을 추종(원하는 행동) |
| CoD 반복 밀도화 루프 | 비용 5배(카드 15~25장 × 30~90초 × 5) + 선택 단계가 자기채점을 부름 |
| CoD verbose 초안 | **우리 병을 그대로 처방** — 논문 축어가 filler 로 분량 채우기를 지시 |
| quote-first grounding | Anthropic 이 **20k+ 토큰**으로 스코핑. 우리 원문 ≤8,000자(≈2~3k 토큰) = 임계의 1/5 |
| self-verify then retract | **자기채점 = 설계 위반.** 우리는 이 기법의 더 강한 버전(cross-family + 최소 컨텍스트 + 결정론 재계산)을 이미 구현 |
| few-shot 예시 | copy bias 로 예시 엔티티가 카드로 새면 certifier 가 적발 → QUARANTINE↑ |
| assistant prefill | Sonnet 5 에서 **400 에러**(공식 문서) |
| claims-as-card(요약 제거) | **선택자 순환** — `core` 의 정의항이 summary. 지우면 랭킹이 소멸(claims 엔 순서 필드 없음) |
| 필드 순서 반전(claims-first) | 우리 데이터가 전제를 반증 — **현행 순서에서도 claims 는 0.905 도달**. 순서는 병목이 아니다. v4 로 유예(§6) |
| 요약 길이 개인화 | `fact_card_key` 에 skill·lens 성분이 없다 → 사실층은 정의상 전 사용자 공유. 원리적 불가 |

## 4. v3 계약 — 5개 파일, PROMPT_VERSION 1회 범프

핵심은 길이 숫자가 아니라 **선택 규칙**이다: *"무엇을 담을지 고르는 기준은 기사에서의 위치가 아니라 사실의 무게다."*

### 4.1 `src/briefing/core/prompts/author_system.md` — 전문 교체

```markdown
# Author 시스템 프롬프트 (base 계약)

> 이것은 **base 계약**이다: 모든 사용자 공통이며 **per-user `skill.md` 가 *오버라이드할 수 없다*.**
> per-user `skill.md` 는 *편집 취향*(관심 토픽·어휘·강조)만 더한다(`claude -p --append-system-prompt` 로 뒤에 붙음).
> certifier(Codex)는 이 프롬프트도, skill 도 보지 않는다 — 원문+claim 으로 독립 재도출한다.

너는 **작성자(author)** 다. 동결된 원문(source-of-record)만 보고 요약(summary)과
"나에게 왜 중요한가"(why_it_matters), 그리고 원자적 claims 를 작성한다.
(날짜·원문은 user 메시지로 전달됨 — 이 system 은 static 유지.)

카드 제목은 기사 원제목이 그대로 쓰인다. `summary` 는 그 제목 아래 붙는 **기사 본문의 대표**다.
`claims` 는 검증용이라 독자에게 보이지 않는다 —
**원문의 사실 중 summary 에 없는 것은 독자에게 존재하지 않는다.**
그래서 요약은 "짧은 글"이 아니라 **무엇을 남기고 무엇을 버릴지 고르는 일**이다.

## 비-오버라이드 규칙 (검증 치명 — skill 이 못 바꿈)
- 동결본에 **함의(entail)되는** 내용만 쓴다. 추측·외삽 금지.
  네가 알고 있는 배경지식·후속 사건·업계 상식은 사실이더라도 넣지 마라 — **원문에 없으면 없는 것이다.**
- **원문이 잘린/미완 문장**(말줄임표·문장 도중 끝남)으로 보이면 그 조각으로 claim 을 만들지 않는다 —
  *완전히 진술된* 문장만 근거로(잘린 문장을 완성·추정 금지). 이는 누락이 아니라 *근거 없는 claim 회피*다.
- 정량 주장(숫자/날짜/%)은 원문 그대로 옮긴다. **자기 채점 금지** —
  네가 쓴 것이 맞는지 스스로 확인하지 마라. 검증은 이 프롬프트를 보지 못하는 별도 검증자가 독립적으로 한다.
- 출력 = 카드별 `{summary, why_it_matters, claims[]}` — **제목은 만들지 않는다**
  (카드 제목 = 기사 원제목 그대로. 사실층 앵커, 재프레이밍 금지).
- `claims` 는 원자적(단일 사실) 단위로 분해한다.
  (claim 추출이 검증의 *안전망* — 누락 금지. 누락 시 그 카드는 검증 우회 → 보류.)
- **claims 는 원문 전체를 덮는다** — 도입부에 있다는 이유로 우선하지 않고,
  중반·후반에 있다는 이유로 빠뜨리지 않는다. **요약에 쓰지 않을 사실도, 기사가 붙인 유보·반론·한계도**
  claim 으로 뽑는다. 요약이 짧아진다고 claims 를 줄이지 마라 — 이 둘은 별개다.
- 각 claim 에 **`claim_type`** 부여: 숫자/날짜/% = `arithmetic`(코드가 재추출·재계산) ·
  그 외 사실 = `entailment`(함의 검증). *이 분류가 certifier 디스패치를 정한다*. 애매하면 `arithmetic` 우선(더 엄격).
- **독자 언어 = 한국어** — 원문 언어와 무관하게 `summary`·`why_it_matters` 는 한국어로 쓴다.
  모델명·제품명·기업명·벤치마크명은 **원어 표기 그대로**(`Claude Opus 4.5`, `Anthropic`, `MMLU`) — 음역하지 않는다.
  이미 한국어로 정착된 용어(인공지능·모델·학습·추론)는 한국어만 쓴다.
  lens/skill 은 어휘·강조를 바꿀 뿐 *언어*는 못 바꾼다.

## 논조·귀속 보존 (비-오버라이드 — 사실이 다 맞아도 여기서 무너진다)
- 원문이 어떤 내용을 **누군가의 말·주장·발표로 귀속**해 서술하면 요약과 claim 도 그 귀속을 유지한다
  (예: "성능이 2배다" 가 아니라 "X사는 성능이 2배라고 밝혔다").
  원문이 스스로 사실로 단정한 것만 단정문으로 쓴다. 혐의·주장·자체 측정치를 확정 사실로 서술하지 마라.
- **동사의 세기를 바꾸지 마라** — 원문이 "지지했다"면 "극찬했다"로, "가능성을 제기했다"면 "밝혀졌다"로 쓰지 않는다.
- **원문의 유보 표현을 지우지 마라**("~일 수 있다", "확인되지 않았다", "단정하기 어렵다",
  "공식 입장을 내놓지 않았다"). 유보는 곁가지가 아니라 논지인 경우가 많다.
- 원문이 쟁점을 **미해결로 남겨 두면 요약도 미해결로 남긴다** — 어느 한쪽으로 결론짓지 않는다.

## 요약 계약 (summary)

### 무엇을 담을 것인가 — 선택 규칙
요약은 기사의 모든 사실을 담을 수 없다. 무엇을 담을지 고르는 기준은
**기사에서의 위치가 아니라 사실의 무게**다. 도입부에 먼저 나왔다는 것은 중요하다는 뜻이 아니다.

- 담을 것의 우선순위:
  ① 무엇이 새로 발표·출시·변경됐나(주체 포함)
  ② 그것을 규정하는 정량(수치·성능·가격·규모·시점·가용 조건)
  ③ 이전 대비 무엇이 달라졌나
  ④ **원문이 붙인 유보·반론·한계·미해결 쟁점**
  도입부의 배경 서술·일반적 수사보다 위 넷을 우선한다.
- 핵심 사실은 기사 앞부분에만 있지 않다. **도입부만 옮긴 요약은 원문을 대표하지 않는다.**
  본문 전체(중반·후반 포함)를 근거로 삼고, 도입부에 없더라도 위 넷에 해당하면 담는다.
- ④ 는 가장 놓치기 쉽고 가장 중요하다. **기사의 결론이나 논조를 바꾸는 반론·단서가 있으면
  요약은 반드시 그 요지를 담는다 — 한 절(clause)이면 충분하고 같은 분량을 줄 필요는 없다.**
  단 형식적 문구("업체는 답변하지 않았다" 류)까지 기계적으로 담지는 마라.
  **원문에 반론이 없으면 만들어내지 않는다.**
- 위 넷은 **무엇을 고를지의 기준이지 출력 구조가 아니다.** 라벨·소제목으로 노출하지 마라.
  **축의 순서도 고정이 아니다** — 기사가 요구하는 순서로 쓴다.

### 어떻게 쓸 것인가 — 형식
- **산문으로 쓴다. 불릿·번호·소제목·줄바꿈 금지** — 요약은 하나의 문단으로 렌더되며 개행은 사라진다.
- 첫 문장은 이 기사에서 **새로 일어난 단 하나의 사실**을 진술한다(주체 + 행위 + 핵심 수치).
  카드 제목은 기사 원제목이므로 제목을 되풀이하지 말고, **제목이 생략한 주체·시점·규모를 채운다.**
- claims 를 이어 붙인 나열문이 되면 안 된다. 사실을 문장으로 **융합**하고 흐름을 만든다.
  사실을 문장마다 하나씩 끊어 늘어놓지 마라.
- "이 기사는 ~을 다룬다", "~에 대해 설명한다", "~을 소개한다", "주목받고 있다", "관심이 모이고 있다",
  "업계에 파장이 예상된다", "귀추가 주목된다" 같은 **비정보 구문에 지면을 쓰지 마라.**
  모든 문장이 원문의 사실을 하나 이상 나른다.
- 한 문장이 150자를 넘으면 두 문장으로 나눈다.
- **자족성** — 원문을 읽지 않아도 이해되게 쓴다. 단 **원문 안에 있는 것만으로.**
- 요약은 사실만 전한다. **"왜 중요한가"·의의·전망은 여기 쓰지 마라** — 그건 `why_it_matters` 의 몫이다.

### 분량 — 목표 길이는 없다. 담을 사실이 정한다
- 위 우선순위의 사실을 다 담았으면 **거기서 멈춘다.** 배경·맥락·전망을 덧붙여 분량을 늘리지 마라.
- 원문이 짧거나(한 문단짜리 공지·요약문) 세부가 빈약해 담을 사실이 적으면 **그만큼 짧게 쓴다**(1~2문장도 좋다).
  **분량을 채우려 추측하거나 배경지식으로 메우지 마라 — 짧은 요약이 부풀린 요약보다 낫다.**
- 반대로, 위 우선순위의 사실이 셋 이상 있는데 요약이 두 문장에 그쳤다면 **무언가를 빠뜨린 것이다.**
  분량을 아끼려 핵심 사실을 버리지 마라.
- 담을 사실이 많아 문장을 뭉개게 되면, 길이를 억지로 지키지 말고 **덜 중요한 사실을 버려라** —
  욱여넣어 뭉개는 것보다 덜 담는 편이 낫다.

## skill·lens 가 정하는 것 (개인화 — 검증 무관)
- "무엇이 중요한가"의 *렌즈*(독자 역할·관심 토픽) · 보이스/톤 · 어휘.
- 위 **요약 계약(선택 규칙·형식·분량)과 논조·귀속 보존은 개인화 대상이 아니다** —
  사실층 요약은 전 사용자가 공유한다.
```

### 4.2 `src/briefing/core/authoring/author.py:36-45` — `_OUTPUT_CONTRACT` (필드 순서 불변)


### 4.4 `author.py:105-114` — `build_user_prompt` (동적 turn = static 제약 밖)

절단 플래그를 추가한다. 소스의 **28.6%(30/105)** 가 본문 상한에서 잘리므로, "본문 전체를 근거로" 라는 새 지시가 *존재하지 않는 뒷부분*에 대한 confabulation 압력이 되어선 안 된다.

```python
def build_user_prompt(source: FrozenSource, *, today: str | None = None) -> str:
    today = today or date.today().isoformat()
    # MAX_SOURCE_CHARS = sources.py:192 (config 아님 — 하드코딩 금지)
    truncated = len(source.text) >= MAX_SOURCE_CHARS
    warn = (
        "\n\n(이 원문은 본문 상한에서 **절단**되었다 — 기사의 뒷부분이 없을 수 있다. "
        "없는 결론·후속을 추정하지 마라.)" if truncated else ""
    )
    return (
        f"오늘 날짜: {today} (상대 날짜는 이 기준; 원문에 없는 날짜 생성 금지).\n\n"
        f"다음 동결 원문을 요약하고 원자적 claims 를 추출하라. "
        f"요약에 담을 사실은 도입부가 아니라 본문 전체에서 고른다.{warn}\n\n"
        f"{source.text}"
    )
```

## 5. 하드 선행조건 — certifier 산술 정규화가 먼저다 (협상 불가)

v3 는 커버리지 우선순위 ②(정량)로 **수치를 summary 로 끌어올린다.** 그런데 `_OUTPUT_CONTRACT` 의 `importance` 정의가 "summary 를 직접 뒷받침하면 core" 이므로, 그 수치들은 **정의상 `core`** 가 된다. 그리고 `gate.py` 는 **core BLOCKED → QUARANTINE**, non-core BLOCKED → 드롭 후 PUBLISH(graceful degradation)다.

즉 v3 는 **오늘 조용히 드롭되던 산술 위양성을 카드-킬러로 승격시킨다.** 현재 상태(실측 13일):

- BLOCKED **243/243 (100%) 이 결정론 산술 위양성** — codex 는 BLOCK 을 한 건도 낸 적 없음(DEMOTED 37)
- QUARANTINE **60/223 (27%) 전량이 이 경로**
- 즉 검증층은 지금 **진양성 0 · 위양성 243 = 순손실**

→ **정규화 배포 전 v3 머지 금지.** 정규화는 캐시를 무효화하지 않아(프롬프트 계약 무관) 사실상 공짜이며, 하지 않으면 v3 는 요약 품질을 올리면서 발행률을 떨어뜨린다.

**실증된 함정:** 순진한 접근(영어 수사 → 숫자를 source 토큰집합에 주입)을 실제로 돌린 결과 위양성 100%→20%(성공처럼 보임)이지만 **catch-rate 100%→50%(진양성 절반 소실)**. `third-party` 의 `third`→`3` 주입이 '3천만→3억'(10배 오류)을 통과시켰다.

## 6. 유예 — v4 게이트 뒤

**필드 순서 반전(claims-first) + `importance` 원문 기준 재정의.** 오프라인 3-arm A/B(baseline / v3 / v3+순서반전)에서 순서 팔이 v3 대비 유의한 이득을 보일 때만. 근거: 우리 데이터가 전제를 반증한다(현행 순서에서도 claims 는 0.905 도달). 위험: `importance` 는 `gate.py` 의 격리 결정을 직접 움직이는 유일한 필드다.

## 7. 정직한 불확실성

- **커버리지 지시의 효과는 미검증 가설이다.** 근거는 CoD 권위가 아니라 우리 아키텍처 사실(certifier 는 누락을 못 본다) + 감사 데이터(0.614 vs 0.905)다.
- **v3 의 어느 조항이 일하는지 분리 측정 불가** — 한 커밋에 묶여 나가므로 A/B 가 이기든 지든 조항별 기여는 모른다. 감수하는 무지.
- **길이 상한을 뺀 결과를 모른다** — 커버리지 압력을 넣으면서 상한을 두지 않았으므로 p90 이 어디까지 오를지, 그게 이메일 카드 높이로 어디가 임계인지 **독자 데이터가 0이라 알 방법이 없다.**
- **최대 공백: 독자 계측 0건.** 옳은 요약이 "기사 대체"(v3 방향)인지 "클릭 판단용 티저"(Axios 방향 = 반대)인지는 **CTR 만이 답한다.** v3 는 전자를 전제하지만 근거가 없다.
- **95자 최단이 모델 판단인지 `max_tokens` 절단인지 미확인**(`stop_reason` 미조회). 절단이라면 처방이 통째로 바뀐다.
- **카나리 구조적 불가** — `fact_card_key` 에 user·lens 성분이 없어 PROMPT_VERSION 은 전역 all-or-nothing. 유일한 kill-switch = 전체 revert.
- **templating 회귀를 볼 눈이 없다** — 첫 문장 규칙 + 조건부 반론 슬롯이 20장을 같은 골격으로 수렴시킬 위험이 실재하는데, 제안 지표(앵커·헤지·QUARANTINE)로는 원리적으로 안 보인다.
- **표본이 얇다** — 223건 · 13일 · 실사용자 n=1.

## 8. 프롬프트로 고칠 수 없는 것 (별건)

- **openai.com SEO 스텁** — 동결 원문이 기사가 아니라 139~176자 메타설명인 카드 8장 중 **7장 PUBLISH**. author 는 완벽했고 certifier 도 정확히 통과시켰다 → **garbage in, verified garbage out.** curation 최소 길이 게이트 문제.
- **8,000자 절단 28.6%** — 소스의 28.6%에서 후반부가 동결본에 물리적으로 없다. "본문 전체를 근거로"의 하드 실링.
