> 증분: v1.0(public 무지출 슬라이스, LIVE) 다음. 선행 리서치: SES 메커니즘 6-에이전트 워크플로(`wf_6ce6447a`).
> 관련: v1.0 스펙 `2026-06-28-web-ui-design.md` · v1.0 전달 `docs/deliveries/web-ui-v1.0-delivery.md`.

## 결정 (사용자 승인)

**2단계로 간다.**
- **v1.1a (이 스펙, 지금)** — **raw SES email-identity 검증 + polling** 으로 sandbox 에서 체험 동작. production access·도메인 불필요 → 지금 빌드·검증 가능.
- **v1.1b (나중)** — production access + 발신 도메인(DKIM) 도착 시, **우리 confirm-token 더블옵트인 + 평판 격리**로 앞단 swap(브랜드 메일·완벽 상관·polling 제거). `mode=trial` 의 "검증 후 발송" 코어는 그대로 재사용.

리서치가 확정한 트레이드오프(수용함): raw 검증은 (1) AWS-브랜드 검증 메일(투박), (2) 이벤트 없음 → polling 필수, (3) 검증 신원 누적·계정평판 위험. → **⑤ 운영 보호 가드는 비협상**(아래).

## 흐름 (raw SES, polling)

```
[/setup 폼] ─① POST /trial {email,sources,…}─► webapi Lambda
   가드: 이메일 1h 쿨다운 · 전역 ≤50/일 cap · email/sources 검증
   이미 Success? ─► 검증 skip                  ② 아니면 ses:verify_email_identity(email)
   briefing-trials 행 upsert(status, ttl)        → 📧 AWS "Email Verification Request"
   ③ invoke_agent_runtime(mode=trial, fire-and-return) → accepted
        ▼
[runtime mode=trial] add_async_task 백그라운드:
   ④ poll ses:get_identity_verification_attributes(email)  ~20s 간격, ≤15분
      Success → run_briefing(임시 UserConfig)→Claude→Codex→gate→make_ses_deliver
                published==0 → 폴백 메일(침묵 방지)
      timeout → 종료(미발송)
        ▼  📬 사용자 받은편지함 (검증 후 발행 1통)
```
★ 사용자 클릭(AWS 검증 링크)이 더블옵트인. 생성(LLM)은 **검증 Success 이후에만** → 미클릭 trial 은 LLM 비용 0.

## 🔒 ⑤ 운영 보호 (비협상 — 도메인 격리 전까지)

도메인 분리 발신이 아직 없어 SES 정지(계정 단위)가 ⑤를 죽일 수 있음. 낮은 볼륨+통제로 방어:
- **전역 일일 cap ≤50 검증/일**(DDB 카운터, 초과 시 429) + **이메일당 1h 쿨다운**.
- **공개 광고 X** — 체험은 동작하되 통제된/조용한 노출(데모). 진짜 공개는 v1.1b(도메인 격리) 후.
- 개발/검증은 **moongons@amazon.com(이미 Success)** allowlist 로 → 검증 메일 0·bounce 위험 0.

## 컴포넌트

### A. runtime `mode=="trial"` — `agentcore_runtime.py` (line 71 `scheduled` 블록 뒤 삽입)
- payload `{mode:"trial", email, sources[], depth?, lens?, timezone?, type?, dry_run?, poll_seconds?, poll_max?}`.
- 임시 `UserConfig`(9필드 전부): `id=f"trial-{_slug(email)}"`, `recipient=email`, `type=payload.get("type","ai-news")`,
  `sources=tuple(sources)`, `depth=payload.get("depth","summary")`, `lens=payload.get("lens","general")`,
  `send_hour=0`, `timezone=payload.get("timezone","Asia/Seoul")`, `skill_md=""`. (users/<id>/ 파일 없음.)
- `add_async_task("trial_briefing")` → 백그라운드 스레드 → `finally: complete_async_task`. 즉시 `accepted` yield.
- 백그라운드:
  ```
  ses = boto3.client("ses", region)
  for _ in range(poll_max≈45):                 # ≤15분 (20s×45)
      st = ses.get_identity_verification_attributes(Identities=[email])
      if status==Success: break
      sleep(poll_seconds≈20)
  else: return                                  # timeout — 미발송
  briefs = run_briefing(settings, store, [trial_user], card_cache=card_cache, ledger=None, run_date=rd)
  b = briefs[0]
  if dry: return
  if b.published>0: make_ses_deliver(settings, subject="브리핑 체험 (검증 후 발행)")(b)
  else: _send_fallback(settings, email)         # "오늘 생성 문제 — 내일 자동 재시도" 폴백
  ```
  - **card_cache 재사용**(content-addressed 키에 source/lens/skill/model 포함 → trial 안전 공유, author+codex 비용 절감) · **ledger=None**(원장 비오염).
- `_send_fallback`: SES send_email 1통(고정 HTML) — published==0(QUARANTINE-only/빈 윈도우/에러)에 침묵 대신 안내.

### B. runtime IAM — `deploy_runtime.attach_runtime_extras`
- inline policy 에 **`ses:GetIdentityVerificationAttributes`** 추가(polling). `ses:SendEmail/SendRawEmail`·bedrock·DDB(briefing-*)는 이미 있음 → **그것만 추가**. → runtime 재배포.

### C. webapi `POST /trial` — `app.py`
- 요청 `{email, sources[], depth?, lens?, timezone?}`. 응답 `{status:"verification_pending"|"sending", trial_id?}`.
- 검증기(신규, config.py TODO 일부 구현): email 형식 · `sources ⊆ catalog_keys() ∧ 1≤len≤5`.
- 가드(DDB `briefing-trials`): 이메일 1h 쿨다운(같은 email 최근 행 있으면 429) · 전역 일일 cap(`GLOBAL#<date>` 카운터 atomic `ADD`, ≤50 초과 시 429).
- `ses.get_identity_verification_attributes([email])`: Success 면 verify skip(status="sending"); 아니면 `ses.verify_email_identity(email)`(status="verification_pending").
- `briefing-trials` upsert: `{email(PK), status, trial_count(+1), created_at, ttl=now+1h}`.
- `bedrock-agentcore` data-plane `invoke_agent_runtime(ARN, mode="trial", payload)` fire-and-return(accepted 까지만 drain, ⑤ `lambda_handler` 동형). read_timeout≈25s(webapi Lambda Timeout=30).
- CORS: `app.py` CORSMiddleware `allow_methods` 에 **`POST`** 추가.
- ⚠️ boto3 클라이언트 필요 → app 임포트는 가볍게(lazy boto3, 모듈 최상단 import 금지 — Mangum cold import 보호).

### D. deploy_api.py — IAM·zip·env·table
- **role**: `_ensure_role` 에 inline policy 추가 — `ses:VerifyEmailIdentity`·`ses:GetIdentityVerificationAttributes`(Resource `*`) · `bedrock-agentcore:InvokeAgentRuntime`(runtime ARN+`/*`) · `dynamodb:GetItem/PutItem/UpdateItem`(`table/briefing-trials`). (무지출 불변식은 v1.1 에서 깨짐 — 의도된 확장.)
- **zip**: `_build_zip` 에 **`boto3`** 추가(bedrock-agentcore data-plane client 누락 회피 — ⑤ 경험). `uv pip install --target fastapi mangum pyyaml boto3`.
- **env**: `BRIEFING_RUNTIME_ARN`(.env), `BRIEFING_TRIALS_TABLE=briefing-trials`, `SES_SENDER`, `TRIAL_GLOBAL_CAP=50`, `TRIAL_COOLDOWN_SECONDS=3600`.
- **table**: `briefing-trials`(PAY_PER_REQUEST, PK `email` S, TTL `ttl`) 생성(idempotent).

### E. web — 체험하기 활성 — `Form.tsx` + `api.ts`
- `api.ts`: `postTrial(payload): Promise<{status}>` (POST `${API_BASE}/trial`).
- `Form.tsx`: 이메일 형식·출처≥1 검증 후 **체험하기 버튼 활성**. 클릭 → postTrial → 상태 표시("확인 메일을 보냈어요 — 클릭하면 곧 브리핑이 도착합니다" / 429 시 "잠시 후 다시 시도"). 구독하기는 v1.2 라 계속 disabled.

### ⚠️ v1.1b 가 반드시 하드닝할 잔여 위험 (최종 리뷰 I1·I2 — v1.1a 에선 cap≤50+미광고로 수용)
- **I1 (가장 날카로움) — trial 동시세션 vs ⑤ no-retry**: cap 은 50/**일**이지 동시 아님 → 공격자가 50건을 수초에 발사 = 최대 50 concurrent 15분 세션이 **⑤와 같은 runtime** 점유. ⑤의 invoke(`scheduler/lambda_handler.py`)·EventBridge 둘 다 **retry 0** → 07:00 KST 와 겹친 flood 가 AgentCore 세션 quota 를 소진하면 그날 ⑤ 브리핑이 조용히 죽을 수 있음(저확률: 미광고·스케줄 추측 필요·기본 quota 보통 >50). **하드닝**: (a) trial 을 **별도 runtime** 으로 분리, 또는 (b) ⑤ poke 에 jitter+throttle-retry, 또는 (c) **동시** trial 세션 cap. → **공개/광고 전 필수.**
- **I2 — 무인증 email-bomb 벡터(≤cap)**: raw `verify_email_identity` 라 미검증 주소마다 AWS 가 피해자에게 검증메일 발송. per-IP/CAPTCHA 없음(CORS `*`는 무인증 엔드포인트에 무의미). v1.1b 의 confirm-token(우리 메일·우리 토큰) + per-IP throttle + CAPTCHA 가 제거.
