## 1. 요약 (Executive Summary)

7개의 후보 애플리케이션을 세 가지 심사 관점(judging lens)으로 평가했습니다 — **실용적 수요·구현 가능성(Practical Demand & Feasibility)**, **제약 적합성·반박 심사(Constraint-Fit, Adversarial)**, **양대 도구 활용도(Dual-Tool Leverage)** — 여기에 순위 비교용 기준선(baseline)으로 **뉴스 브리핑 에이전트**를 포함했습니다.

상위 4개 아이디어가 평균 **4.33점**으로 동률을 이뤘습니다 (Obligation Ledger, BriefGuard, TutorProof, Diligence Room). 동률은 평균 점수가 가려버리지만 각 관점이 드러낸 세 가지 기준으로 갈립니다:

1. **현재 하니스 안에서의 구현 가능성** (유료 코퍼스나 규제 의존성이 없을 것).
2. **제약 적합성의 강도** — 특히 지속 가능한 고유 상태(durable proprietary state), **되돌릴 수 없는 실세계 부수효과(irreversible real-world side effects)**, 폐쇄 피드백 루프(closed feedback loop)의 *결합*.
3. **양대 모델 분업이 구조적으로 핵심을 담당하는가** — 기본 제공되는 생성/리뷰 루프를 단지 다시 칠한 수준이 아닐 것.

### 추천: **Obligation Ledger (의무 원장)**

> 체결된 계약 포트폴리오를 의무·기한·갱신/해지 트리거의 살아 있는 모니터링 원장(ledger)으로 전환합니다. **Claude Code**가 각 의무를 추출하고 외부로 보낼 산출물(예: 갱신 거절 통지문)을 작성하며, **Codex**가 날짜/통지 기간 산술을 결정론적 코드로 독립 재도출하고 인용된 조항을 확인합니다 — **어떤 트리거가 실행되기 전에(verify-before-fire).**

이 안이 승리한 이유는, 동시에 *가장 구현하기 쉽고*(PDF 텍스트 + 추출 + JSON/SQLite 원장 + 스케줄러 — 별난 인프라 없음, 유료 코퍼스 없음) *제약 적합성을 가장 깔끔하게 통과*하기 때문입니다: **되돌릴 수 없는 부수효과**(해지 통지 발송, 법정 기한 누락)를 만들어내는 다년치 상태 원장을, 강력한 **verify-before-fire 게이트** 뒤에 둡니다. 이것이 막아주는 실패 모드 — 30일 어긋난 통지 기간이 잘못된 해지 통지문을 조용히 발송하는 것 — 는 정확히 단일 생성 모델이 과신하고, 상관관계가 끊긴 두 번째 모델 **+ 실행 가능한 날짜 산술**이 진가를 발휘하는 영역입니다.

## 2. 제약 해석 및 포함 테스트 (Constraint Interpretation & Inclusion Test)

### 제약을 다시 정의하면

핵심 가치가 **Claude Code와 Codex를 함께 결합**한 상호보완적 멀티 하니스 엔진에 의존하며, **지속 가능한 외부 도메인**(고유 상태, 실세계 부수효과, 폐쇄 피드백 루프) 위에서 작동하고, **단일 하니스·MCP 서버·Skill·스케줄러가 기본 제공으로는 이미 해내지 못하는** 것이어야 하는 **실용적** 애플리케이션을 만든다.

### 포함 테스트 (Inclusion test — 후보는 아래 모두를 통과해야 함)

- **두 하니스가 핵심을 담당해야 함.** Codex를 제거(또는 한 모델이 스스로 점검하는 형태로 축소)하면 중심 신뢰 속성(central trust property)이 — 단지 마감 품질이 떨어지는 게 아니라 — 본질적으로 깨져야 한다.
- **지속 가능한 외부 도메인.** 제품이 고유하고 진화하는 상태(원장, 데이터룸, 학생별 프로필)를 소유한다 — 일회성 리포트가 아니다.
- **기본 프리미티브를 넘어설 것.** Skill + WebSearch/WebFetch + `/schedule` + PushNotification을 쌓아서 재현되지 않는다.

### 제외 대상 제품 유형 (*핵심*으로 삼으면 자동 탈락)

뉴스/콘텐츠 브리핑 에이전트; 범용 딥리서치/리포트 작성기; 코드 생성·코드 리뷰·PR 어시스턴트; 크론/리마인더 실행기; 스크레이퍼/브라우저 봇; Gmail/Calendar/Drive 어시스턴트; 문서 조회 도구; 알림 디스패처; AWS 스캐폴딩 헬퍼; Skill/hook 생성기; **얇은 LLM 래퍼**(텍스트에 대한 요약/분류/추출/재작성); 멀티 에이전트 프레임워크.

### 승자가 기준을 통과하는 방식

Obligation Ledger는 **얇은 추출 래퍼가 아닙니다**: 지속 가치는 항목이 **실제로 되돌릴 수 없는 조치**를 실행하는 **다년치 의무 원장**이며, 이를 **트리거 로직을 재도출하는 독립된 두 번째 모델**이 게이트합니다. `/schedule`은 폴링할 수 있지만, 상태를 가진 원장과 의무별 판정 대조(verdict reconciliation)가 **곧 제품 자체**입니다 — 크론 틱(cron tick)은 부차적입니다.

## 3. 라우팅 전략 — Claude Code 대 Codex

**토폴로지: Specialized (역할 분리).** Claude Code는 오케스트레이터/생성자(orchestrator/generator)이고, Codex는 **차단(block)** 권한과 1회의 제한적 재생성을 트리거할 권한을 가진 전담 독립 검증자(independent verifier)입니다. 이는 Relay(독립 교차 점검이 없어 생성자 오류가 곧장 실세계 부수효과로 흘러감)와 Ping-Pong(지연(latency) + 두 모델이 같은 오답으로 수렴할 위험)보다 우월합니다. 신뢰 속성은 **직무 분리(separation of duties)** 에서 나옵니다: *산출물을 작성하는 모델과 그것을 승인하는 모델이 다르다.*

| 파이프라인 단계 | 도구 | 이 도구인 이유 |
|---|---|---|
| **수집(Gather)** — 인테이크, 계약 템플릿별로 어떤 의무 유형을 찾을지 결정 | **Claude Code** | 개방형 계획 수립: 계약군 + 설정된 Skill을 구체적 추출 계획으로 분해. 에이전트형 정문(front door)을 담당. |
| **수집(Gather)** — 계약 PDF 파싱/정규화, 조항 분절, 정의된 용어 해소 | **Both(양쪽)** | Claude가 오케스트레이션하며 Skill로 정의된 의무 스키마를 적용; Codex가 결정론적 파싱·OCR 정리·상호참조 해소 코드를 작성/실행. |
| **생성(Generate)** — 각 의무 레코드 + 외부 산출물(예: 갱신 거절 서신) 초안 작성 | **Claude Code** | 섬세한 지시 이행, 계약 전반의 장문 컨텍스트 종합, 브랜드/관할권에 맞는 작성. |
| **검증(Verify)** — 날짜/통지 기간 산술 재도출; 인용 구간이 주장된 조건을 담는지 확인 | **Codex** | **실행 가능한 코드**로 독립적·반박적 재도출하여 생성자의 조용한 날짜 산술 오류와 상관관계를 끊음. 치명적 오류 부류가 모델 추측이 아닌 결정론적 판정이 됨. |
| **검증(Verify)** — JSON 스키마 검증, 완전성 점검, 영업일/공휴일 캘린더 규칙 | **Codex** | 결정론적 PASS/FAIL 게이팅 신호를 오케스트레이터가 실행 전에 차단 근거로 사용. |
| **전달(Deliver)** — 리마인더 예약, 산출물 렌더링·승인 대기 큐 적재, 감사 레코드 기록 | **Claude Code** | 사용자 대면 계약과 엔드투엔드 실행을 담당; Codex가 만든 렌더러/검증기를 하위 단계로 호출. |

**Claude Code 강점:** 모호한 의도 분해, 브랜드/톤 준수, 장문 컨텍스트 종합, skill/hook 오케스트레이션, 엔드투엔드 실행 소유.
**Codex 강점:** 상관관계 끊긴 반박적 검증, 정밀한 결정론적 파싱/연산/검증 코드, 깔끔한 PASS/FAIL 게이트, 중대한 산출물의 신뢰를 높이는 두 번째 모델 감사.

## 4. 후보 비교 (뉴스 브리핑 기준선 대비 순위)

관점별 점수: **PDF** = 실용적 수요·구현 가능성 · **CF** = 제약 적합성(반박) · **DTL** = 양대 도구 활용도.

| 순위 | 후보 | PDF | CF | DTL | 평균 | 기준선 대비 | 한 줄 평 |
|---|---|---|---|---|---|---|---|
| **1** | **Obligation Ledger** | 4 | 4 | 5 | **4.33** | **+2.33** | 다년치 의무 원장; Codex가 통지 기간 산술을 코드로 재도출하고 **어떤 트리거가 실행되기 전에** 조항을 확인. 가장 구현 쉽고, 되돌릴 수 없는 부수효과가 가장 명확. **승자.** |
| 2 | BriefGuard (법률 인용/판시 검증기) | 4 | 4 | 5 | 4.33 | +2.33 | Codex가 인용된 판례의 존재·유효 법(good law)·명제 뒷받침 여부를 독립 점검. 환각 인용 부류를 코드로 제거; good-law 모니터링이 해자(moat)지만 유료 citator 필요. |

| 4 | Diligence Room | 4 | 4 | 5 | 4.33 | +2.33 | Codex가 모든 실사(diligence) 발견을 인용 구간에 grounding한 뒤에야 메모가 go/no-go를 좌우. 강력하나 RAG가 무겁고 딥리서치에 근접. |

**모든 후보가 기준선보다 위에 랭크됩니다.** 기준선은 **제약 적합성에서 1점**입니다 — 전체 파이프라인이 기본 프리미티브에 1:1로 매핑되고, 유일한 차별점(지속 가능한 출처 신뢰/정정 원장)이 구현된 것이 아니라 가설에 그치기 때문입니다. 상위 4개는 어떤 단일 하니스도 제공하지 않는, 중대한 산출물에 대한 강력한 교차 모델 게이트 + 지속 가능한 고유 상태를 근거로 +2.33만큼 앞섭니다.

## 5. 심층 분석 — Obligation Ledger

### 5.1 문제 (Problem)

기업은 수백 건의 계약(공급사 MSA, 임대차, NDA, SOW)을 체결한 뒤 그것을 **조용히 위반**합니다: 자동 갱신이 흘러가고, 통지 기간이 닫히고, SLA 크레딧을 청구하지 못하고, 면책 한도(indemnity cap)를 잊습니다. 위험은 **일회성 추출이 아니라** — 원장을 *수년에 걸쳐* 정확하게 유지하는 것입니다. 한 번 잘못 추출된 "통지 기간"이나 "자동 갱신일" 하나가 실제 기한 누락이나 잘못된 해지 통지를 유발하기 때문입니다. 범용 추출 도구는 독립된 두 번째 점검 없이 날짜와 조항 상호참조를 환각합니다.

### 5.2 대상 사용자 (Target user)

여섯 자릿수 비용의 CLM 플랫폼(Ironclad/Icertis/Evisort)을 정당화하기 어려운 **중견기업의 사내 법무 운영(legal ops)/상사 계약 관리자**. 이들은 이미 계약을 수작업 검토로 처리하고 있으며, 지속 가능하고 신뢰할 수 있는 의무 등록부에 대해 명확하고 반복적인 지불 의향을 가집니다.

### 5.3 엔드투엔드 파이프라인 (누가 무엇을 하는가)

```
        ┌─────────── CLAUDE CODE (오케스트레이션 / 생성 / 전달) ───────────┐
        │                                                                  │
 계약서 │  [수집]            [생성]                       [전달]           │
 PDF ──▶│  파싱 + 계획  ──▶  의무 레코드 초안 작성  ──┐    리마인더 예약    │
        │  Skill별 의무      {당사자, 트리거,         │   + 통지문 렌더링   │
        │  유형 정의         기한 계산, 근거 구간,    │   (승인 대기 큐)    │
        │                    권장 조치}               │   + 감사 로그       │
        └────────────────────────────────┬───────────┴──────────▲─────────┘
                                          │ 원본 근거 구간 +      │ PASS
                                          │ 제안 레코드           │ (원장 기록 /
                                          ▼ (최소 컨텍스트)       │  실행)
        ┌──────────── CODEX (독립 검증 / 게이트) ─────────────────┴─────────┐
        │  • 지배 조항을 독립적으로 재탐색                                   │
        │  • 날짜 산술을 "실행 가능한 코드"로 재도출                         │
        │      (effective_date + 30일 통지, 12개월 갱신 전 / 영업일 vs 일반) │
        │  • 인용된 구간이 실제로 주장된 조건을 담는지 확인                  │
        │  • JSON 스키마 + 완전성 검증                                       │

          지속 상태(Durable state): 다년치 의무 원장 (JSON/SQLite)
          신규 계약/수정 계약 유입 시 재-diff  ── 폐쇄 루프(closed loop)
```

### 5.4 교차 모델 검증 단계 (두 모델이 핵심을 담당하는 이유)

두 가지 실패 모드가 두 도구에 깔끔하게 매핑됩니다:

- **추출 오류** (Claude가 잘못된 조항을 지목, 통지 단위 오독) → Codex가 지배 조항을 **독립적으로 재탐색**하고 해당 구간이 주장된 조건을 담는지 확인하여 포착.
- **추론 오류** (30일 어긋난 산술, 영업일 vs 일반일 착오, 롤링/에버그린 기간) → Codex가 **결정론적 코드로 산술을 재도출**하여, 트리거 로직이 모델 추측이 아니라 *실행 가능하고 감사 가능한 연산*이 되도록 함.

결정적으로, Codex에는 Claude의 초안 레코드나 근거가 아니라 **최소 컨텍스트**(원본 구간 + 스키마)만 제공됩니다. 작성자의 추론을 숨기면 확인 통과(confirmation pass)가 아니라 **독립적 재도출**이 강제됩니다 — 이것이 실제로 오류 상관관계를 끊습니다. 추출과 검증을 모두 하는 단일 모델은 같은 맹점을 공유하여, 상관된 오류가 곧장 되돌릴 수 없는 부수효과(발송된 해지 통지)로 흘러가게 둡니다. **Codex를 제거하면 중심 신뢰 주장이 붕괴합니다.**

> **게이트에 내장된 설계 가드레일** (심사자들의 주요 위험 지적에 대응):
> 1. Codex는 Claude의 구간을 신뢰하지 말고 조항을 **독립적으로 탐색**해야 한다 — 그렇지 않으면 둘 다 같은 잘못된 입력을 두고 추론한다.
> 2. 되돌릴 수 없는 조치의 기본 **ACTION-MODE는 초안 작성 후 승인 대기 큐**다; 자율 실행은 의무 유형별 옵트인. 잘못된 통지가 통과되는 verify-before-fire 게이트는 자동화 없음보다 나쁘다.
> 3. 격리된(모호하나 참일 수 있는) 의무는 **사람 플래그**로 강등될 뿐, 조용히 폐기되지 않는다.

### 5.5 Skill 기반 커스터마이징 표면 (type / brand / depth 대응)

Skill이 기준선의 TYPE/BRAND/SUMMARY-DEPTH 노브에 대응하는 것을 설정합니다:

| 노브 | 대응 | 설정 내용 |
|---|---|---|
| **CONTRACT-FAMILY** | TYPE | 임대차 vs SaaS MSA vs 고용 vs NDA — 어떤 의무 스키마와 조항 패턴을 적용할지 결정. |
| **OBLIGATION-TYPES** | (범위) | 추적 대상: 갱신, 통지 기간, SLA/크레딧, 면책 한도, 독점, MFN. |
| **JURISDICTION** | BRAND/로캘 | 영업일 vs 일반일 관례, 공휴일 캘린더, 지역별 날짜 산술. |
| **ACTION-MODE** | DEPTH | 알림만 · 통지문 초안 · 초안 후 승인 대기 — 부수효과가 어디까지 갈지 제어. |

### 5.6 MVP 범위 (오늘, 하니스 안에서 구현 가능)

**범위 내(v1):**
- **하나의** 계약군에 대해 깔끔한 텍스트/양호한 OCR PDF를 인제스트(시작: SaaS MSA — 갱신/통지 가치가 큼).
- Claude가 의무 레코드 {당사자, 트리거, 기한 산술, 근거 구간, 권장 조치}를 추출 → JSON/SQLite 원장.
- Codex **독립 검증기**: 조항 재탐색, 날짜 산술 코드 재도출, 구간-조건 포함 점검, 스키마 검증 → 강제 PASS/FAIL.
- `/schedule`(CronCreate/Monitor)이 리마인더 리드타임을 구동; 임박 트리거에 PushNotification/SendMessage.
- v1에서는 **ACTION-MODE = 초안 후 승인 대기만**(자율 실행 없음). Claude가 갱신 거절/통지 산출물을 사람 승인용으로 작성.
- 의무별 **감사 레코드**: 출처, 두 모델의 판정, 승인자 — 지속 가능하고 재-diff 가능한 상태.

**범위 외(추후):** 스캔/필기 OCR 파이프라인, 다중 문서 수정 계약 체인, 자율 실행, SOC2/테넌시 격리, 광범위한 다중 계약군 템플릿.

**노력 포지션:** PDF 텍스트 + Claude 추출 + Codex 검증기 + 원장 파일 + 스케줄러. 웨어하우스 없음, 라이선스 코퍼스 없음, 모델 학습 없음. 어려운 엔지니어링은 가치가 있는 곳 — verify-before-fire 게이트와 지속 가능한 원장 상태 머신 — 에 정확히 집중됩니다.

## 6. 솔직한 위험 및 승리 이유 (Honest Risks & Why It Wins)

### 뉴스 브리핑 기준선을 이기는 이유
기준선은 **명시적으로 제외 집합**에 있고(Skill + WebSearch/WebFetch + `/schedule`이 약 90%를 재현) **제약 적합성 1점**입니다. 포함 기준을 통과시키는 유일한 요소인 출처 신뢰 원장은 가설에 불과합니다. 반대로 Obligation Ledger는 **진짜 다년치 고유 상태**를 소유하고 강력한 교차 모델 게이트 뒤에서 **되돌릴 수 없는 실세계 부수효과**를 만들며, 이는 기본 프리미티브가 분명히 제공하지 못합니다. 격차는 평균 +2.33, 제약 적합성에서 무려 3점입니다.

### 다른 상위 4개 동률을 이기는 이유
- **vs BriefGuard / Diligence Room:** 둘 다 **유료 법률 코퍼스**(good-law 판단을 위한 KeyCite/Shepard's) 또는 기밀이며 증가하는 코퍼스에 대한 **무거운 RAG**에 게이트됩니다. Obligation Ledger의 진실 원천(고객 자신의 체결 계약)은 **이미 손안에** 있습니다 — 페이월 없음, 거짓 "존재하지 않음" 실패를 낳는 커버리지 공백 없음.
- **vs TutorProof:** 마찬가지로 구현이 쉽고 교과서적 양대 도구 적합성이지만, 검증의 강점이 **계산 수학에 집중**됩니다(서술형/증명의 기호적 동치 점검은 취약). Obligation Ledger의 고가치 연산(통지 기간 날짜 산술)은 **완전히 결정론적이고 판정 가능**하여, 게이트가 정확히 중요한 곳에서 빈틈없습니다.

### 솔직한 위험 (및 완화책)
1. **책임의 역전.** 잘못 발송된 통지 한 건이 제품이 막으려는 바로 그 치명적 피해를 유발. → **완화:** 기본 ACTION-MODE는 사람 승인; 자율 실행은 유형별 옵트인.
2. **혼잡한 카테고리** (Ironclad, Icertis, Evisort, DocuSign). 차별점이 검증 게이트인데 이는 *마케팅하기* 어려움. → **완화:** CLM이 없는 **중견기업**을 타깃; 풀 CLM이 아니라 인용 검사기로 포지셔닝.
3. **공유 맹점 / 수렴.** Codex가 Claude의 레코드를 보면 앵커링이 동의 쪽으로 끌고, 프리미티브(발효일, 통지 단위) 추출은 여전히 LLM 작업. → **완화:** Codex에 최소 컨텍스트(원본 구간 + 스키마)만 주고, 이미 틀린 프리미티브에 대한 산술 재점검이 아니라 **독립적 조항 재탐색**을 요구.
4. **지저분한 실제 PDF** (스캔본, 수정 계약, 정의된 용어 상호참조)가 진짜 병목이며, 두 모델이 OCR/분절 맹점을 공유할 수 있음. → **완화:** v1을 한 계약군의 깔끔한 텍스트 PDF로 한정; 낮은 신뢰도 추출은 조용히 신뢰하지 말고 사람 검토로 노출.
5. **엔터프라이즈 보안/영업 주기.** 기밀 계약 코퍼스는 벤더 보안 검토를 요구. → **완화:** 빌드가 아닌 GTM 비용으로 인정; v1 크리티컬 패스 밖.
6. **라벨링된 코퍼스 없이는 검증이 측정 불가.** Codex가 중요한 오류를 잡는지, 단지 도장만 찍는지 증명 없음. → **완화:** 알려진 오추출의 소규모 적대적 테스트셋을 v1 수용 게이트로 구축; 게이트 존재가 아니라 포착률(catch-rate)을 보고.

## 7. 차순위 후보 (간단히)

### BriefGuard — 법률 인용·판시 검증기 (평균 4.33)
서면 제출 전, Claude가 각 법적 명제 + 인용을 추출하고 뒷받침 논거를 작성; Codex가 인용된 판례의 **존재**, **유효 법(good law) 여부**, **주장된 명제를 실제로 뒷받침하는지**를 독립 검증 — 결정론적 존재/형식 점검을 강제 게이트로 산출하고, 판시 왜곡은 변호사 검토로 플래그. **환각 인용이라는 치명적 부류를 코드로 제거**(후보 중 가장 강력한 단일 안전 서사)하며, 해자는 인용 판례의 후속 이력이 바뀜에 따라 **시간에 걸쳐 재-diff되는** 사안별 권위 원장. **왜 1위가 아니라 2위인가:** 권위 있는 good-law 상태가 **유료 citator**(KeyCite/Shepard's) 뒤에 있음; 무료 코퍼스는 존재는 주지만 신뢰할 만한 폐기(overruled) 신호는 주지 못해, 핵심 good-law 모니터링이 v1에서는 최선 노력 수준. 또한 단일 서면 정적 점검은 딥리서치 + WebFetch에 근접.

### TutorProof — 정답 검증형 적응 연습 엔진 (평균 4.33)
Claude가 개인화된 연습 문제와 풀이를 생성하고 학생별 장기 숙달 프로필을 유지; Codex가 Claude 풀이를 보지 않고 **처음부터 실제 연산을 돌려 각 문제를 독립 재풀이**하고, 학생에게 도달하기 전에 결정론적으로 정답을 대조. 거의 교과서적 양대 도구 적합성 — "모델이 자기 채점을 한다"는 정확한 실패 모드를 상관관계 해소 — 이며, 순수 인프라(문제 생성 + 재풀이 + 코드 실행 + 학생별 JSON 상태 파일) 기준 가장 구현이 쉬움. **왜 차순위인가:** 정답 동치는 의외로 어렵고(1/2 vs 0.5, 인수분해형 vs 전개형, 증명, 서술형), 신뢰할 만한 검증이 주장된 "대수1~MCAT" 전 범위가 아니라 계산 수학/논리로 좁아짐; FERPA/COPPA가 학생별 프로필에 비엔지니어링 범위를 더함.

### Diligence Room (평균 4.33)
딜의 증가하는 데이터룸을 인제스트; Claude가 실사 체크리스트를 조항 단위 인용으로 답하고 레드플래그 등록부를 구축; 모든 발견을 독립 Codex grounding으로 라우팅하여, 실사 메모가 밸류에이션이나 go/no-go를 좌우하기 전에 어떤 주장도 무근거가 되지 않게 함. 강력한 지속 상태(비공개·증가 코퍼스 + 영속 레드플래그 등록부)지만, RAG/코퍼스 처리가 더 무겁고 제외된 딥리서치 레인에 근접.
